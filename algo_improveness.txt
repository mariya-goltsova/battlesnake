ALGO APPROACH — ПЛАН ДОВЕДЕНИЯ ВЕТКИ algo ДО ПОБЕДНОГО СОСТОЯНИЯ
================================================================

ЦЕЛЬ
----
Ветка algo — чистая, предсказуемая эвристика. Она должна стать «слоем безопасности
и объяснимости»: сильная, быстрая, без магии. Её же логику безопасности и симуляции
переиспользует гибрид (см. unite_algo_ml.txt). Задача: убрать грубые упрощения
(статичные хвосты, Manhattan, жадный 1-шаг) и сделать скоринг калибруемым.

ЧТО УЖЕ ЕСТЬ (факты по коду, logic.py ветки algo)
-------------------------------------------------
- choose_move -> choose_move_heuristic (logic.py:47-49). Жадный перебор 4 ходов.
- Для каждого легального хода:
    score = _flood_fill(nxt, occupied, limit=my_length+1)         # место
    if nxt in danger: score -= 10_000                             # h2h штраф
    if hungry and foods: score += (W+H - manhattan_nearest)*2     # подкормка
  argmax, иначе "up" (logic.py:100-101).
- Вспомогательные: _occupied_cells (все сегменты + хвосты вечны, logic.py:104),
  _head_to_head_cells (клетки рядом с головами >= нашей длины), _flood_fill (DFS,
  капнут на my_length+1), _manhattan, _in_bounds.

ГЛАВНЫЕ ПРОБЛЕМЫ ИМЕННО ALGO-ВЕТКИ
----------------------------------
1. Статичная занятость. _occupied_cells держит хвосты вечно занятыми -> бот считает
   безопасные проходы (вдоль своего/чужого хвоста) непроходимыми и сам себя загоняет.
2. Manhattan до еды. (W+H-nearest)*2 не учитывает стены из тел и здоровье реально:
   можно ломиться к «близкой» еде через тело или туда, куда враг придёт первым.
3. flood_fill капнут на my_length+1 (logic.py:85) — считает «влезу ли я», но не даёт
   сравнить простор двух безопасных ходов (оба вернут limit). Нужна и полная оценка.
4. Один скор-скаляр. Место + штраф + еда слиты в одно число, стратегию нельзя
   калибровать по компонентам.
5. Жадность на 1 ход. Нет lookahead -> самозапирания и шаг в вилку h2h.
6. Нет оценки территории (выходы, choke, степень свободы после хода) и нет
   enemy-race-to-food.

ПЛАН РАБОТ (по приоритету; P0 = сделать в первую очередь)
--------------------------------------------------------

P0-1. Корректная модель занятости на следующий ход.
   Что: simulate_after_move(state, move):
     - двигаем свою голову; убираем свой хвост, если nxt НЕ еда (при еде растём —
       хвост остаётся); аналогично освобождаем хвосты противников, если они не едят.
   Где: заменить прямое использование _occupied_cells в choose_move_heuristic и
     в _flood_fill-контексте на «occupied после симуляции».
   Зачем: перестаём считать проходимые коридоры стенами; больше валидных ходов и
     меньше самозапираний. Общая функция для algo/ml/гибрида.
   Оценка: 0.5–1 день. Impact: высокий.

P0-2. BFS/A* до еды вместо Manhattan.
   Что: для каждой еды считать кратчайший БЕЗОПАСНЫЙ путь (BFS по свободным клеткам
     на симулированной карте). Учитывать health: идти к еде агрессивнее, если
     bfs_food_dist близко к health; не ломиться к недостижимой еде.
   Где: заменить блок logic.py:92-94 (hungry-подкормка).
   Зачем: корректная пищевая экономика — ключ к выживанию в solo и к росту в 1v1.
   Оценка: 3–4 часа. Impact: высокий.

P0-3. h2h как правило, а не только штраф.
   Что: если есть хотя бы один ход без h2h-риска — не выбирать h2h-клетку вовсе
     (hard veto). Штраф 10_000 оставить только как tie-break, когда все ходы плохи.
   Где: choose_move_heuristic — фильтровать danger-клетки при наличии альтернативы.
   Зачем: убирает главный источник глупых лобовых смертей.
   Оценка: 1–2 часа. Impact: высокий.

P1-1. Разделить скоринг на компоненты.
   Что: считать отдельно space_score, food_score, danger_score, center_score,
     attack_score; итог = взвешенная сумма с ЯВНЫМИ весами-константами вверху файла.
   Зачем: стратегию можно калибровать и объяснять; те же компоненты станут фичами
     для ml/гибрида (переиспользование).
   Оценка: полдня. Impact: средний-высокий (разблокирует калибровку).

P1-2. Оценка территории (не только размер простора).
   Что: помимо flood_fill добавить:
     - число выходов из nxt (свободные соседи),
     - точки сочленения / choke points (после хода не остаёмся с 1 узким выходом),
     - «степень свободы» = полный flood_fill (без cap) для сравнения двух безопасных
       ходов; сравнивать с my_length (влезаем ли целиком).
   Зачем: перестаём заходить в «мешки», которые формально проходимы, но тупиковые.
   Оценка: 0.5–1 день. Impact: высокий.

P1-3. One-step / two-step lookahead.
   Что: минимакс глубины 2: наш ход -> «худший разумный» ответ противников -> оценка
     эвристикой. Отсекать ходы, ведущие в форсированную смерть на 2-м тике.
   Где: minimax(state, depth) с бюджетом времени; при таймауте — жадный 1-шаг.
   Зачем: главный шаг качества после исправления карты и еды.
   Оценка: 1 день. Impact: высокий.

P1-4. enemy_race_to_food.
   Что: BFS от голов противников до каждой еды; если враг приходит первым — не
     тратить ходы на эту еду (или идти на перехват, если мы длиннее и можем прижать).
   Оценка: 3–4 часа. Impact: средний-высокий.

P2-1. Тестовые состояния поля (обязательно, идут параллельно P0/P1).
   Что: набор фиксированных board-состояний + ожидаемое поведение:
     стена; собственный хвост (должен проходить после P0-1); еда рядом; h2h с длинной
     змеёй (уклон); h2h с короткой (можно идти/атаковать); тупик (не входить);
     коридор (пройти вдоль хвоста); отсутствие safe moves (fallback не падает).
   Где: tests/ (pytest), состояния как JSON.
   Зачем: любая правка скоринга не должна ломать базовую безопасность.
   Оценка: полдня стартово, дополняем по ходу. Impact: высокий (страховка).

P2-2. Eval-харнесс (общий с ml — см. ml_improveness.txt / unite_algo_ml.txt).
   Что: скрипт гоняет сотни/тысячи партий через Battlesnake CLI и считает:
     win-rate, среднюю длину жизни, причины смерти (wall/body/h2h/starvation),
     p95 latency. Гонять algo против ml и против чужих бейзлайнов.
   Зачем: без цифр нельзя доказать, что правка улучшает, а не ухудшает.
   Оценка: 0.5–1 день. Impact: высокий.

ОПРЕДЕЛЕНИЕ ГОТОВО / МЕТРИКИ
---------------------------
- Проходят все tests/ (базовая безопасность).
- В харнессе: solo — стабильно 300+ тиков без самосмертей; 1v1 против прежнего
  algo-бейзлайна win-rate > 60%; p95 latency < 200ms (эвристика должна быть быстрой).
- Скоринг разбит на именованные компоненты с явными весами.

РИСКИ
-----
- Слишком агрессивный food/attack -> голодные/лобовые смерти: держать P0-3 veto и
  тесты P2-1.
- Рост latency от lookahead: бюджет времени + fallback на жадный ход (P1-3).

ПОРЯДОК ДЕЙСТВИЙ (короткий): P0-1 -> P0-2 -> P0-3 -> тесты/харнесс -> P1-1 -> P1-2
-> P1-3 -> P1-4 -> P2. Все P0-функции (симуляция, BFS-еда, h2h-veto) писать в общем
модуле, чтобы гибрид переиспользовал их без переписывания.
