# Battlesnake: симулятор игры + поэтапный RL (всё в src_rl/)

## Правила работы

- **Весь код, тесты, планы и артефакты — только внутри `src_rl/`. Корень репозитория не изменяется**
  (`backend.py`, `logic.py`, `requirements.txt`, `render.yaml` остаются как есть).
- **Никаких git-операций из этой сессии** (checkout/commit/branch/stash): в корне репозитория
  параллельно работает вторая Claude-сессия (ветка `stage0-hybrid-core`, план
  `docs/plans/stage0-hybrid-core.md` — рефакторинг корня). Файлы `src_rl/` остаются untracked
  и не конфликтуют ни с одной веткой; коммит src_rl — отдельное решение пользователя.

## Контекст

Корневой бот: Flask (`backend.py`) + `logic.py`, ходы ранжирует линейная модель по 13 признакам
(flood fill, Voronoi/BFS, h2h-опасность, еда), веса зашиты литералами в `_MODEL` —
невоспроизводимы. Симулятора, обучения и тестов в корне нет. Известные баги признаков
(из заметок `*_improveness.txt`): еда по Манхэттену «сквозь стены», хвосты считаются вечно
занятыми, h2h — мягкий штраф вместо жёсткого запрета, нет lookahead.

**Можно ли сделать симуляцию и применить RL — да.** Правила открыты, свой движок ~300–400 строк;
для обучения есть 128 CPU-ядер, A100 80GB, Python 3.11.

**Решения:** поэтапно (Этап 1 — симулятор + подбор весов линейной модели self-play'ем;
Этап 2 — deep RL PPO на том же симуляторе); режим — стандарт 11×11, 4 змеи (работает и в 1v1);
serving остаётся лёгким (чистый Python, numpy — только на Этапе 2), p95 < 300 мс.

## Структура пакета

```
src_rl/
├── RL_PLAN.md                  # этот план
├── __init__.py
├── core.py                     # примитивы: DIRECTIONS, in_bounds, bfs_dist, flood_fill, occupied_cells
├── sim.py                      # полный движок правил (step, init_game, make_game_state) + simulate_after_move
├── safety.py                   # legal_moves (с учётом хвостов), жёсткий фильтр (стена/тело/h2h), least_bad_move
├── features.py                 # ЕДИНАЯ функция признаков (паритет train == serve)
├── policy.py                   # пайплайн: фильтр → симуляция → признаки → скоринг из model.json → fallback
├── model.json                  # экспортированные веса {feature_names, mean, std, coef, meta}
├── backend_rl.py               # ОТДЕЛЬНЫЙ Flask-вход для нового бота (корневой backend.py не трогаем)
├── baselines/legacy_policy.py  # замороженная копия текущего logic.py (оппонент + baseline)
├── train/
│   ├── selfplay.py             # play_game + multiprocessing-раннер (120 ядер)
│   ├── opponents.py            # пул: legacy, эвристика, random-safe, снапшоты
│   ├── es_train.py             # CMA-ES цикл по весам
│   ├── evaluate.py             # турнир: win-rate ± CI, причины смертей, латентность
│   ├── export.py               # запись model.json + parity-проверка
│   └── rl/                     # Этап 2 (env.py, ppo.py, distill.py) — позже
├── tests/                      # unit/property/differential + fixtures/
├── tools/                      # официальный battlesnake CLI (в src_rl/.gitignore)
└── requirements-train.txt      # numpy, cma, pytest (serve-зависимости корня не трогаем)
```

Импорты: `python -m src_rl.train.es_train` из корня; тесты: `python3 -m pytest src_rl/tests -q`.

**Serving нового бота без изменения корня:** `src_rl/backend_rl.py` — свой Flask-app на тех же
4 эндпоинтах, импортирует `src_rl.policy.choose_move`. Локально/CLI гоняем его на отдельном
порту. Переключение деплоя Render (правка render.yaml — корень!) — отдельное решение
пользователя после Этапа 1.

## Этап 0 — ядро + P0-фиксы (каждый шаг проверяем pytest'ом)

1. Каркас `src_rl/` + этот план. ✔
2. `baselines/legacy_policy.py` — копия текущего `logic.py` (замораживается навсегда) +
   фикстуры досок + характеризационные тесты `_flood_fill`/`_bfs_dist`/`_candidate_features`.
3. `core.py` — общие примитивы (bfs_dist, flood_fill, occupied_cells и т.д.) + тесты.
4. `sim.py` — полный движок правил (см. ниже) + `simulate_after_move(game_state, move)`:
   своя голова двигается, свой хвост сохраняется если съели; хвосты врагов освобождаются,
   если рядом с их головой нет еды (консервативно). Тонкость: после еды `body[-1] == body[-2]`
   один тик — хвост НЕ освобождается.
5. `safety.py`: `legal_moves` с учётом хвостов; жёсткое h2h-вето (если остаётся хоть один
   не-h2h ход); `least_bad_move` вместо «up и надейся». Тесты на фикс-досках.
6. `features.py`: признаки на симулированном состоянии; еда через `bfs_dist` (недостижимая →
   сентинел); новые признаки `food_reachable`, `bfs_food_dist`, `length_advantage`.
7. `policy.py` + `backend_rl.py`; текущие веса `_MODEL` → стартовый `model.json`
   (миссcalibrated на новых признаках — приемлемо, жёсткий фильтр страхует, Этап 1 перетюнит).
8. Тест латентности: p95 `choose_move` < 50 мс локально (запас ×6 к бюджету Render 300 мс).

## Движок правил (`sim.py`) — порядок тика как в официальном engine

1. Двигаем всех живых змей (новая голова, отбрасываем хвост).
2. `health -= 1`.
3. Кормление: голова на еде → `health = 100`, дублируем хвостовой сегмент; едят ВСЕ, кто попал.
4. Спавн еды: `len(food) < minimum_food` → доспавнить; иначе с вероятностью
   `food_spawn_chance/100` (15%) одна еда на случайной свободной клетке. RNG сидируется.
5. Элиминация в две фазы: A — голод, выход за границы; B (одновременно для выживших после A) —
   self/body-collision, head-to-head (короче — умирает, равные — оба; длины ПОСЛЕ кормления).
6. `make_game_state(state, snake_id)` выдаёт точный формат API → одни и те же policy-функции
   работают и вживую, и в симуляции.

**Валидация** — дифференциальные тесты против официального CLI (github.com/BattlesnakeOfficial/rules,
бинарь в `src_rl/tools/`, версия зафиксирована): записываем игры `battlesnake play ... --output
game.jsonl`, восстанавливаем ходы по смещению голов, прогоняем через наш `step()`, сравниваем.
Точное сравнение — прогоны с `--foodSpawnChance 0`; обычные — по модулю новой случайной еды.
~50 игр — в фикстуры. Плюс property-тесты (змеи не пересекаются, health∈[0,100],
length == len(body)...) и юнит-тест на каждое правило.

## Этап 1 — self-play подбор весов (CMA-ES)

- `train/selfplay.py`: `play_game(policies, seed, cfg)` → winner/ranks/turns/причины смертей.
  `multiprocessing.Pool(120)`, батчи по ~8 игр, политики как picklable-спеки (не замыкания!),
  запуск скриптами из терминала (`__main__` guard), не из ноутбука. Ожидаемо ~4–8 тыс. игр/мин.
- Оптимизируем ~16 весов (13 старых + 3 новых признака). mean/std пересчитать один раз по
  ~100k состояний из baseline-игр и ЗАМОРОЗИТЬ. θ₀ = текущие коэффициенты + нули.
- CMA-ES (pip `cma`; запасной вариант — свой CEM ~50 строк). Популяция 24–32;
  fitness = 128 игр (96 четырёхзмеечных + 32 1v1) против пула (legacy, эвристика, random-safe
  + снапшоты каждые K поколений). Очки: победа 1.0, ранги в 4p (0.4/0.15/0), +0.001·туры.
- **Общие сиды внутри поколения** (common random numbers) — парное сравнение, режет шум в разы.
- Бюджет: ~3.6k игр/поколение ≈ 1–2 мин → 150–300 поколений за ночь.
- Отбор модели НЕ по train-fitness: каждые 10 поколений — eval-харнесс, берём лучшую по eval.
- `train/evaluate.py`: 1000 игр 1v1 против каждого фиксированного бенчмарка (95% CI ≈ ±3.1%),
  4p mean rank, разбивка причин смертей, латентность. Общие сиды + парный тест.
- `train/export.py` → `model.json`; parity-тест: скорер serve-пути и train-пути дают идентичный
  argmax на фикстурах (защита от train/serve-дрифта).

**Done Этапа 1:** симулятор проходит все тесты включая дифференциальные; тюнингованный гибрид
бьёт legacy в 1v1 с win-rate ≥ 60% (CI исключает 50%); ноль провалов в safety-сьюте «никогда
не выбирает избегаемую смерть»; латентность в бюджете; `python -m src_rl.train.es_train --seed`
воспроизводит сопоставимый `model.json`.

## Этап 2 — deep RL (эскиз, детально планируем после Этапа 1)

- `train/rl/env.py`: Gymnasium-среда поверх `sim.py`; оппоненты — замороженные политики из
  пула внутри env (сэмплируются на эпизод).
- Наблюдение: **эгоцентрический тензор C×11×11** (тело с «тиками до освобождения», головы
  врагов длиннее/короче, еда, health/длина как broadcast-планы).
- PPO с маскированием недопустимых действий из `safety.py` (маска в rollout и в лоссе),
  CleanRL-стиль. Роллауты на 128 CPU, апдейты маленькой CNN (<200k параметров) на A100.
- Стабильность self-play: пул = 50% свежих чекпоинтов + 50% история + постоянные якоря
  (гибрид Этапа 1, эвристика); eval только против фиксированных якорей.
- Награда: терминально +1/−1 (+ранговый кредит в 4p), +0.002/тур, +0.1 за еду при health<30.
- Экспорт: веса CNN → `.npz`/JSON + чистый numpy forward (~100 строк, единицы мс).
  Жёсткий safety-фильтр и fallback — всегда сверху.
- Done: ≥60% 1v1 против гибрида Этапа 1 (CI исключает 50%), safety-сьют зелёный, латентность в бюджете.

## Ключевые риски

- **Расхождение симулятора с реальными правилами** → дифференциальные тесты против CLI
  закреплённой версии; настройки ruleset читать из запроса.
- **Переобучение self-play** → лига оппонентов со снапшотами и якорями; отбор по held-out eval.
- **Шумный fitness глушит CMA-ES** → общие сиды, эскалация числа игр в конце, ранговый кредит.
- **Multiprocessing в Jupyter** → только терминальные скрипты, спеки вместо замыканий.
- **Параллельная сессия в корне** → никаких git-операций; работаем только в src_rl/.

## Верификация (end-to-end)

1. `python3 -m pytest src_rl/tests -q` — юнит/property/дифференциальные/parity/латентность.
2. Смоук serve: `python -m src_rl.backend_rl` (отдельный порт) + POST фикстуры в `/move`.
3. Игра через официальный CLI: новый бот (`backend_rl`) vs legacy (`backend.py`) на двух портах.
4. `python -m src_rl.train.evaluate` — таблица win-rate ± CI: tuned vs legacy vs эвристика vs random-safe.
