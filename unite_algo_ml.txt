UNITE ALGO + ML — ГИБРИД КАК ПОБЕДНАЯ СТРАТЕГИЯ ХАКАТОНА
========================================================

ГЛАВНЫЙ ВЫВОД
-------------
Не «ML вместо правил» и не «правила вместо ML», а конвейер:
    жёсткие правила безопасности (из algo) -> ML-ранжирование безопасных ходов (из ml).
algo даёт безопасность, предсказуемость и объяснимость; ml — более умный выбор среди
уже безопасных ходов. Такой гибрид сильнее каждой ветки по отдельности и его не стыдно
показать на демо (можно объяснить каждое решение).

ПОЧЕМУ ГИБРИД, А НЕ ЧИСТЫЙ ML/ALGO
----------------------------------
- Чистый ml опасен как боевая стратегия: линейная модель может (при плохой калибровке
  большого voronoi) перебить мягкий h2h-вес и увести под лобовое / в тупик. На демо
  «модель иногда самоубивается» — это провал.
- Чистый algo предсказуем, но потолок эвристики ниже: он не «чувствует» тонкий выбор
  между двумя одинаково безопасными ходами так же хорошо, как обученная модель.
- Гибрид: правила гарантируют, что мы НИКОГДА не выбираем заведомо смертельный ход,
  а модель выбирает лучший из выживающих. Худший случай гибрида не хуже algo.

ЦЕЛЕВОЙ КОНВЕЙЕР (pipeline)
---------------------------
    Battlesnake request
        v
    parse game_state
        v
    generate candidate moves (up/down/left/right)
        v
    HARD SAFETY FILTER  (из algo):
        - вне поля -> удалить
        - в тело (с учётом симуляции хвостов) -> удалить
        - hazard-смерть -> удалить
        - заведомо проигранный h2h при наличии альтернативы -> удалить
        v
    SIMULATE NEXT STATE для каждого кандидата:
        - двигаем свою голову; хвост освобождается, если не съели; при еде растём
        - аккуратно двигаем/освобождаем хвосты противников
        v
    FEATURES (из ml, на симулированной карте):
        - flood fill (capped и full open space)
        - Voronoi (контроль территории)
        - tail reachability (доступ к своему хвосту = не заперты)
        - BFS distance to food (НЕ Manhattan) + food_reachable + health
        - enemy race to food (успеваем ли раньше врага)
        - h2h risk / near_bigger_head / length_advantage
        - escape routes / choke points / wall_dist / dist_to_center
        v
    ML / WEIGHTED SCORER: ранжирует ТОЛЬКО прошедшие фильтр ходы
        v
    TIE-BREAKERS + FALLBACK:
        - при равенстве — больше простора/центр/доступ к хвосту
        - если после фильтра пусто -> «наименее смертельный» ход (max flood_fill)
        v
    return move

АРХИТЕКТУРА КОДА (чтобы ветки реально слились, а не жили параллельно)
--------------------------------------------------------------------
Ключ к объединению — общий код, а не копипаста. Обе ветки уже используют ОДНИ И ТЕ ЖЕ
базовые функции (_occupied_cells, _head_to_head_cells, _flood_fill, _bfs_dist,
_manhattan, _in_bounds). Выделить их в модули:

    core.py     — типы, DIRECTIONS, _in_bounds, _bfs_dist, _flood_fill, _manhattan
    sim.py      — simulate_after_move(state, move): корректная занятость (P0 обеих веток)
    safety.py   — legal_moves(), hard_safety_filter(): стена/тело/hazard/h2h-veto
    features.py — ОДНА функция фич, импортируемая и train.py, и logic.py (паритет!)
    policy.py   — choose_move: filter -> simulate -> features -> score -> tie-break/fallback
    model.json  — экспортированные mean/std/coef (или дерево), инференс pure-python

logic.py остаётся тонкой обёрткой над policy.choose_move. backend.py не меняется по сути
(добавить только бюджет времени / защиту latency).

ВАЖНО: три «общих кита» (simulate_after_move, BFS-до-еды, h2h-veto) — это ровно те P0,
что запланированы в algo_improveness.txt и ml_improveness.txt. Если писать их сразу в
общих модулях, объединение получается почти бесплатно.

ПЛАН СЛИЯНИЯ ПО ЭТАПАМ
---------------------
Этап 0 (подготовка, параллельно обеим веткам):
  - Вынести общее ядро в core.py/sim.py/safety.py/features.py.
  - Реализовать P0 обеих веток В ОБЩИХ модулях: simulate_after_move, BFS-еда, h2h-veto.
  - Поднять eval-харнесс (win-rate, длина жизни, причины смерти, p95 latency) и
    tests/ с фиксированными board-состояниями.
  Готово, когда: algo и ml оба зовут общие модули; тесты зелёные; харнесс считает цифры.

Этап 1 (собрать гибрид):
  - policy.py: hard_safety_filter -> simulate -> features -> scorer(model) -> fallback.
  - Скорер сначала = существующая линейная модель ml (она уже pure-python), но теперь
    ранжирует только безопасные ходы на корректных фичах.
  Готово, когда: гибрид в харнессе >= max(algo, ml) по win-rate и не хуже по latency.

Этап 2 (усилить мозг):
  - Воспроизводимое обучение (train.py + features.py + датасет self-play) — из ml P1-1.
  - 2-ply lookahead поверх safety-filter — из обеих веток P1.
  - Новые фичи: enemy_race_to_food, choke/articulation, length_advantage, trapped_enemy.
  Готово, когда: гибрид стабильно бьёт бейзлайны и держит p95 latency < 300ms.

КАК СРАВНИВАТЬ (обязательно измеримо)
------------------------------------
Единый харнесс гоняет турнир: algo vs ml vs hybrid vs чужие бейзлайны, N партий на
каждую пару, разные размеры поля (7x7, 11x11) и режимы (solo, 1v1, 4-player). Метрики:
win-rate, средняя длина жизни, разбивка смертей (wall/body/h2h/starvation), p95 latency.
Решение о мёрдже принимаем ТОЛЬКО по цифрам харнесса, не «на глаз».

КРИТЕРИИ ПОБЕДЫ (definition of done для хакатона)
------------------------------------------------
- Гибрид никогда не выбирает заведомо смертельный ход при наличии безопасного
  (доказано тестами P2 из algo_improveness.txt).
- Win-rate гибрида > каждой из веток по отдельности в харнессе.
- p95 latency < 300ms на Render free (иначе таймаут = смерть).
- Есть воспроизводимое обучение (train.py + датасет) — можно защитить метрику на демо.
- Есть демонстрируемый нарратив: «правила гарантируют выживание, модель выбирает лучший
  из выживающих ходов» + графики из харнесса.

ПОРЯДОК ДЕЙСТВИЙ (короткий):
  Этап 0 (общее ядро + P0 + харнесс/тесты) -> Этап 1 (собрать pipeline, линейный скорер)
  -> Этап 2 (train.py, 2-ply, новые фичи). Приоритет ресурсов: сначала безопасность и
  измеримость, потом «ум» модели.
